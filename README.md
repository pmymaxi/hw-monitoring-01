## Системы мониторинга

##### Ответ на вопрос 1:

> Вас пригласили настроить мониторинг на проект. На онбординге вам рассказали, что проект представляет из себя платформу для вычислений с выдачей текстовых отчетов, которые сохраняются на диск. Взаимодействие с платформой осуществляется по протоколу http. Также вам отметили, что вычисления загружают ЦПУ. Какой минимальный набор метрик вы выведите в мониторинг и почему?

Для определения проблемы касающейся загрузки ЦПУ в момент вычислений, применим следующие метрики: 
1.	CPU Load Avarage для понятия загрузки системы и ее очереди в соответствии с количеством ядер системы. 
2.	Так как после вычисления отчеты сохраняются в файл на диск, применим метрику ввода и вывода диска I/O и FS, данной метрикой мы сможем понять имеется ли очередь на чтение или запись на диск и на сколько переполняется или используется пространство файловой системы. 
3.	Также применим метрику, например top 10 высоконагруженных процессов, сможем определить источник загрузки системы, а также определим количество дочерних потоков.  
4.	 Метрика RAM/Swap, данной метрикой определим общею утилизацию RAM и скидывает он ли в swap память, если скидывает, то это существенно моет замедлять работу сервиса. 
5.	Метрики HTTP (RPS количество запросов, время ответа, количество ошибок http запросов 4хх/5хх). Взаимодействие с платформой осуществляется по HTTP поэтому нужно знать общею картину по запросам и ошибкам. 
При таком наборе метрик достаточно, чтобы определить проблему, и понять связанна она из-за постоянной максимальной утилизации системы или в проблеме работы http сервиса. 

##### Ответ на вопрос 2:

> Менеджер продукта посмотрев на ваши метрики сказал, что ему непонятно что такое RAM/inodes/CPUla. Также он сказал, что хочет понимать, насколько мы выполняем свои обязанности перед клиентами и какое качество обслуживания. Что вы можете ему предложить?

Можно предложить в добавлении метрики уровня сервиса (SLI):
-	Availability сервиса, для определения процента доступности сервиса 
-	Затраченное время обработки запроса, можно определить на сколько быстро клиенты получают результат на запрос. 
-	Количество успешных обработанных запросов относительно общего числа запросов 
Также для этих метрик можно определить SLO

##### Ответ на вопрос 3:

> Вашей DevOps команде в этом году не выделили финансирование на построение системы сбора логов. Разработчики в свою очередь хотят видеть все ошибки, которые выдают их приложения. Какое решение вы можете предпринять в этой ситуации, чтобы разработчики получали ошибки приложения?

Решение из полноценного стека сервисов: 
-	Fluent Bit агент сбора и отправки логов на целевой сервер 
-	OpenSearch система хранения, поиска и обработки логов 
-	OpenSearch Dashboards web сервис для визуальной обработки логов. 
Если нужны только ошибки, можно применить бесплатный SaaS Sentry перехватчик ошибок. 

##### Ответ на вопрос 4:

> Вы, как опытный SRE, сделали мониторинг, куда вывели отображения выполнения SLA=99% по http кодам ответов. Вычисляете этот параметр по следующей формуле: summ_2xx_requests/summ_all_requests. Данный параметр не поднимается выше 70%, но при этом в вашей системе нет кодов ответа 5xx и 4xx. Где у вас ошибка?

Ошибка применении формулы расчета, в расчете не применяется количество 3xx запросов, так как они являются тоже успешным кодом для http. Правильная формула расчетов 
(summ_2xx_requests + summ_3xx_requests) / summ_all_requests

##### Ответ на вопрос 5:

> Опишите основные плюсы и минусы pull и push систем мониторинга.

Push метод
Плюсы:
- Упрощение репликации данных в разные системы мониторинга или их резервные копии
- Более гибкая настройка отправки пакетов данных с метриками
- UDP — это менее затратный способ передачи данных, из-за чего может возрасти производительность сбора метрик
Минусы:
- Сложнее контролировать доступность агентов 
- Возможны потери метрик при использовании UDP; 
- Требуется настройка отправки на каждом агенте; 
- Сложнее централизованно менять параметры конфигурации сбора и отправки метрик.
- В не которых системах мониторинга (например Zabbix) следить за соответствием версии агента относительно к серверу. 

Pull метод
Плюсы:
- Легче контролировать подлинность данных
- Можно настроить единый proxy server до всех агентов с TLS
- Упрощённая отладка получения данных с агентов
- Централизованное управление сбора метрик
Минусы:
- Сложности при работе за NAT и динамически создаваемыми узлами 
- Нагрузка на сервер при большом сборе метрик с целевых хостов/приложений
- Доступ сервера ко всем агентам
